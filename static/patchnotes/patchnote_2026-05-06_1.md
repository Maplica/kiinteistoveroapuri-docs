# Patchnote — 2026-05-06 #1

## Overview
This patch improves row interaction in both the left and right table windows of the Veroselvitysavustaja dialog. The main areas of change are multi-row selection, right-click copy functionality, an improved "Näytä kartalla" zoom button that handles multiple selections, and several stability fixes that were discovered and resolved during implementation.

---

## New Features

### Multi-row selection in both table windows
Both `tableView_features` (left) and `tableView_addedFeatures` (right) now support selecting multiple rows simultaneously using standard keyboard modifiers:
- **Ctrl + click** — add or remove individual rows from the selection
- **Shift + click** — select a contiguous range of rows

Previously both tables were set to `SingleSelection`, which only ever allowed one row to be highlighted at a time. Both are now set to `ExtendedSelection` with `SelectRows` behaviour so that clicking any cell in a row highlights the full row, and multiple rows can be held selected at once.

The mutual exclusivity between the two tables is preserved: selecting a row in the left table automatically clears any selection in the right table and vice versa. This is enforced by the existing `_on_features_selection_changed` and `_on_added_features_selection_changed` handlers, which were left untouched.

---

### Right-click "Kopioi solun arvo" on the left table
A new context menu appears when right-clicking any cell in `tableView_features`. It contains a single action:

- **Kopioi solun arvo** — copies the display value of the clicked cell to the system clipboard.

The menu resolves which cell was under the cursor at the time of the right-click using `indexAt(pos)`, maps the proxy index to the source model index, reads the `QStandardItem` text, and pushes it to `QApplication.clipboard()`. If the cursor is not over any cell (e.g. below the last row), the action is shown but disabled.

This is implemented in the new method `_on_features_context_menu(self, pos)`.

---

### Right-click "Kopioi solun arvo" on the right table
The existing right-click context menu on `tableView_addedFeatures` (previously only containing session save/load actions) has been extended with a copy action at the top:

- **Kopioi solun arvo** — copies the display value of the clicked cell, using `addedProxyModel.mapToSource()` to access the underlying source model.
- A visual separator line separates the copy action from the session management actions below it.
- Session actions (Tallenna / Tallenna nimellä… / Lataa…) are unchanged.

---

### "Näytä kartalla" supports multiple selected rows
The `handle_pushButton_zoomTo_clicked` method has been fully rewritten to support any number of selected rows in either table.

**Single feature selected** → behaviour unchanged: the map is centered on that feature at a fixed 1:1000 scale and a red locate highlight is drawn.

**Multiple features selected** → the new `_zoom_to_multiple_features` method is called:
1. Collects the bounding box of every selected feature, transforming each geometry to the canvas CRS.
2. Unions all bounding boxes into one combined extent.
3. Applies 20 % padding on all sides (minimum 50 map units) so no feature sits right at the viewport edge.
4. Sets the canvas extent to the padded rectangle and refreshes.
5. Draws a red locate highlight on every selected feature simultaneously.

The rewrite also removed a legacy bug where the method used `currentIndex()` to detect which table had focus. `currentIndex()` can linger in Qt even after a selection is cleared, making it unreliable. The new version uses `selectedRows()` on both tables — whichever returns a non-empty list is used, with no priority ordering needed because the exclusivity guarantee means only one table can be non-empty at a time.

---

### `_add_locate_highlight` — new `clear_first` parameter
A `clear_first=True` keyword argument was added to `_add_locate_highlight`. When `True` (the default), all existing locate highlights are cleared before adding the new one — this preserves the previous behaviour for all single-feature callers. When `False`, new highlight markers are appended without wiping earlier ones, which is required by the multi-feature loop in `_zoom_to_multiple_features`.

---

### New `_zoom_to_multiple_features(features_list)` method
A new method that accepts a list of `(layer, fid)` tuples and zooms the map canvas to show all of them at once. Full docstring and inline comments document every step: bounding box collection, CRS transformation, extent union, padding calculation, canvas update, and highlight loop.

---

### Inline editing disabled on the right table
`tableView_addedFeatures` now has `setEditTriggers(NoEditTriggers)` applied. Previously the table inherited Qt's default edit triggers, which allowed double-clicking directly into a cell (including the Kommentti column) and typing into it. This caused a QGIS crash because the `_on_added_item_changed` signal handler tried to interact with a partially-constructed edit state. All editing of any field, including Kommentti, must now go through the double-click row dialog (`_handle_added_row_double_clicked`), which validates and persists changes safely.

## Bug Fixes

### NameError: `selected_indexes` not defined in `handle_pushButton_moveToAdded_clicked`
**Cause:** During the rewrite of `handle_pushButton_zoomTo_clicked`, the code replacement accidentally consumed the first line of the immediately following method `handle_pushButton_moveToAdded_clicked`. The line `selected_indexes = self.tableView_features.selectionModel().selectedRows()` was removed, leaving the `for proxy_index in selected_indexes:` loop referencing an undefined name.

**Fix:** The missing line was restored as the first statement of `handle_pushButton_moveToAdded_clicked`.

---

### Access violation crash when moving multiple rows to the right table
**Cause:** `handle_pushButton_moveToAdded_clicked` iterated directly over the list of proxy `QModelIndex` objects returned by `selectedRows()`. Mid-loop, the body called `featuresModel.setItem(...)` to paint the `•` (dot) marker in the last column. This modification to the source model triggers `proxyModel` to recompute its internal row-to-source mapping, which **invalidates the remaining proxy indexes** still in the iteration list. Accessing an invalidated Qt C++ `QModelIndex` causes an access violation. With single selection the loop always ended after one iteration before any invalidation occurred, so the bug was invisible until multi-row selection was introduced.

**Fix:** The method was refactored into two phases:
1. **Resolve phase** — iterate over all selected proxy indexes and call `proxyModel.mapToSource()` for each, collecting the resulting source indexes into a plain Python list. No model modifications occur here.
2. **Process phase** — iterate over the already-resolved source indexes and perform all the existing logic (store entry creation, status update, dot marker). Since these are source model indexes and the proxy mapping is no longer consulted, invalidation of the proxy has no effect.

## Code Quality

All new and modified methods received:
- Full **docstrings** documenting purpose, parameters, return values, and call sites.
- **Inline comments** on non-obvious logic blocks explaining the reasoning in plain language, not just restating the code.
- Short comments on changed setup lines explaining why the change was made (e.g. why `ExtendedSelection` replaces `SingleSelection`, why `NoEditTriggers` was added, why the two-phase resolve approach is necessary).

Finnish UI-facing strings (menu labels, tooltips) remain in Finnish. Code-level comments are in English.
