# Patchnote — 2026-05-26 #1

## Overview

This patch fixes the **Building_Combined green dots** rendering bug caused by Verohallinto renaming a source CSV column, and makes two small UX/quality-of-life improvements.

---

## Bug Fix

### Building_Combined rendered all features as green "else" dots

**Root cause:** Verohallinto renamed `VRK-rakennustyyppi` → `Ryhti pääasiallinen käyttötarkoitus / VRK-rakennustyyppi` in the source CSV data (2025+).  After `tag_columns_with_origin()`, the column arrives in the GeoPackage as:

```
Ryhti pääasiallinen käyttötarkoitus / VRK-rakennustyyppi__vero
```

But all Building_Combined QML styles use `classAttribute="VRK-rakennustyyppi__vero"`.  No features matched any category, so every feature fell through to the "else" symbol (the green dot).

**Fix — `_STABLE_COLUMN_REGISTRY` in `3.0_Tiedonkasittely_to_exe.py`:**

A new registry maps canonical output column names (what QML, analysis_dialog, and analysisutils expect) to lists of known Verohallinto source-name variants.  At the end of processing, before writing each layer to the GeoPackage, any column whose name matches a known alias is renamed to the canonical form.

```python
_STABLE_COLUMN_REGISTRY: dict[str, list[str]] = {
    'VRK-rakennustyyppi__vero': [
        'Ryhti pääasiallinen käyttötarkoitus / VRK-rakennustyyppi__vero',  # 2025+
    ],
    # … 13 more __vero columns pre-listed as inventory (empty alias lists) …
}
```

When Verohallinto renames a column again in a future year:
1. Append `'new_name__vero'` to the relevant list in `_STABLE_COLUMN_REGISTRY`.
2. Rebuild the exe.
No QML, plugin, or `_VERO_COLUMN_ALIASES` changes are needed.

**Files changed:** `Kiinteistö_Vero/exe creation/3.0_Tiedonkasittely_to_exe.py`

## Code Quality

### CSV/tietue reading pipeline refactored to Phase 1 placeholder approach

The rakennusosa extra-column fix (introduced in patchnote_2026-05-12_1) has been redesigned. The previous implementation handled the fix in **Phase 3** with special-case detection and a drop step. The current implementation moves the adjustment to **Phase 1**, producing a fully uniform Phase 3 loop.

**Before (Phase 3 drop approach):**
Phase 3 sliced one extra column of headroom for rakennusosa, then compared the actual column count to the expected count and conditionally dropped the extra column before assigning headers. All other record types used a straightforward slice.

**After (Phase 1 placeholder approach):**
When `rakennusosa_fix_enabled` is `True`, Phase 1 inserts a placeholder column name (`_extra_kolumni`) at position `rakennusosa_extra_col_index` when building the expected-headers list for `df_rakennusosa`. This increases `_n` by one, so Phase 3's slice `group.iloc[:, 1:_n+1]` picks up exactly the right number of columns — including the extra one — without any branching. Phase 3 is now identical for every record type:

```python
_data = _group.iloc[:, 1:_n + 1].copy()
if _data.shape[1] != _n:
    _dlog(f"  [VAROITUS] Sarakemäärä {_data.shape[1]} ≠ odotettu {_n} — ohitetaan")
    continue
_data.columns = _expected_cols
```

The `_extra_kolumni` placeholder travels through the pipeline and appears in the output GeoPackage as `_extra_kolumni__vero`. It contains whatever raw value the CSV had at that column position and does not affect any QML, analysis, or report logic.

Config keys `rakennusosa_fix_enabled` and `rakennusosa_extra_col_index` are unchanged. The Asetukset dialog UI is unchanged.

**Files changed:** `Kiinteistö_Vero/exe creation/3.0_Tiedonkasittely_to_exe.py`

### Removed legacy missing-category injection workaround

The inline missing-category injection code that was added as a temporary workaround for the green-dots rendering bug has been removed now that the root cause is fixed.

**Removed from `Prosessointi_dialog.py`:**

- `_inject_missing_categories()` static method — was never called (dead code)
- Inline injection block in `apply_qml_styles_to_layers()` — injected orange→blue colour-ramped categories for any data value not already in the QML renderer; no longer needed
- `unique_values_cache` dict — was only used by the injection block
- `QgsCategorizedSymbolRenderer`, `QgsRendererCategory`, `QgsGradientColorRamp`, `QgsSymbol`, `QColor`, `QVariant` imports — were only needed by the removed injection code
- Stale docstring references to the two-pass approach and missing-category optimisations

**Files changed:** `veroselvitysavustaja/kiinteistoveroApuri/Prosessointi_dialog.py`
