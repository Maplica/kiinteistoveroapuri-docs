# Patchnote — 2026-05-20 #1

## Overview

This patch introduces a **Rekisterikorjaus** workflow — a new third status category (Status = 2) for features that are recognised as register-correction candidates rather than genuinely missing tax-authority records.  The workflow spans the QGIS layer data model, the right-click context menu in the main window's left table, a new report type, an expanded report generation dialog, and updated QML styles that render the new status visually.

---

## New Features

### Status = 2 — "Rekisterikorjaus" (Register Correction)

A new feature status is introduced alongside the existing **0 = Check** and **1 = OK**:

| Value | Label | Meaning |
|---|---|---|
| 0 | Check | Potential discrepancy — requires review |
| 1 | OK | Verified as correct |
| **2** | **Rekisterikorjaus** | Key mismatch; not a missing record — internal register correction needed |

Rekisterikorjaus is intended for cases where a feature from the municipality register cannot be automatically matched to a tax-authority record because of a key mismatch (e.g. PRT or kiinteistötunnus discrepancy), but the feature itself **does exist** in the tax authority data.  These features must **not** appear in the "puuttuvat" report sent to Vero — marking them Status = 2 keeps them out of all external reports entirely.

### ~~Right-click "Merkitse rekisterikorjaukseksi" in the left table~~ *(removed)*

> **Update:** This right-click action was subsequently removed from the plugin. The section below is kept for historical reference only.

The existing right-click context menu on `tableView_features` has been extended with a second action for supported layers:

| Action | Available |
|---|---|
| Kopioi solun arvo | Always (when a cell is selected) |
| **Merkitse rekisterikorjaukseksi** | Only for Rekisterikorjaus-capable layers |

**How it works:**

1. The user right-clicks any cell in the left features table.
2. If the currently loaded layer is in `_REKISTERIKORJAUS_LAYERS`, the new action is enabled.
3. Selecting it calls `_mark_feature_as_rekisterikorjaus(layer, fid)`, which writes `Status = 2` directly to the feature in the QGIS layer (starts editing, commits immediately).
4. The map canvas is refreshed so the colour change is visible immediately.
5. The feature is **not** added to the right-window (`_added_features_store`) and will **not** appear in any report.

**Supported layers** (`_REKISTERIKORJAUS_LAYERS`):

- `Building_Database_data_not_in_Taxdata (prosessoitu)`
- `Property_To_Check (prosessoitu)`

### New report type — Rekisterikorjaus CSV (internal)

A second family of CSV reports has been added alongside the existing "puuttuvat" reports.  Rekisterikorjaus reports are **for internal use only** and contain no contact block.

#### `create_rekisterikorjaus_df()` — `util/reportingutils.py`

New public function that produces a tidy, per-field-change DataFrame from a list of session store entries.

**One row is produced per changed field per entry.**  If an entry has three edited fields, three rows appear in the output.

**Column set — `category='rakennus'`:**

| Column | Description |
|---|---|
| Kiinteistötunnus | From stable session keys (fallback: `id_field` in row_data) |
| PRT | Permanent building identifier from session keys |
| Rakennuksen numero | Building number from session keys |
| Kenttä | Display name of the changed field (internal suffix stripped) |
| Alkuperäinen arvo | Value before editing |
| Uusi arvo | Value after editing |
| Kommentti | User comment saved with the session entry |

**Column set — `category='maapohja'`:** Same but without PRT and Rakennuksen numero.

**`side` parameter** controls which suffix group is included:

- `'kunta'` → only `__kunta` field changes
- `'vero'` → only `__vero` and `__added` field changes

Fields that are present in `tax_data` but whose value is identical to `row_data` are silently skipped.  Empty DataFrames still carry the correct column structure.

### `_collect_report_entries()` — entry routing into 6 buckets

A new method that partitions all entries in `_added_features_store` into six named buckets:

`rakennus_puuttuvat` · `rakennus_kunta` · `rakennus_vero` · `maapohja_puuttuvat` · `maapohja_kunta` · `maapohja_vero`

Routing uses two independent steps applied in sequence for every entry:

**Step A — layer-name routing**
If the entry's `origin_layer_name` matches a known processed layer name in `_LAYER_BUCKET_MAP`, it is placed in the corresponding bucket.  Currently mapped layers:

| Layer name | Bucket |
|---|---|
| `Rekisterin_rakennukset_puuttuvat_verotiedosta (prosessoitu)` | `rakennus_puuttuvat` |
| `Veroaineiston_rakennukset_puuttuvat_rekisteristä (prosessoitu)` | `rakennus_kunta` |
| `Rekisterin_Kiinteistöt_puuttuvat_verotiedosta (prosessoitu)` | `maapohja_puuttuvat` |

**Step B — edit-side routing (`_entry_edit_sides`)**
If `tax_data` contains real changes (different from `row_data`), the changed field suffixes determine which buckets receive the entry:
- `__kunta` field edits → `{prefix}_kunta`
- `__vero` or `__added` field edits → `{prefix}_vero`

**Fallback:** If neither step matched anything, the entry goes to `{prefix}_puuttuvat`.

An entry can appear in multiple buckets (e.g. one that has both `__kunta` and `__vero` edits).

## Bug Fixes

### CRS-valinta: muokattu koordinaattijärjestelmä ei aiheuta enää kaatumista

**Ongelma:** Jos käyttäjä valitsi koordinaattijärjestelmäksi *muokatun koordinaattijärjestelmän* (`QgsProjectionSelectionWidget` palautti CRS:n, jonka `authid()` on `""` tai `"USER:XXXXX"`), prosessointiohjelma kaatui kryptiseen CRS-virheeseen, koska pyproj/geopandas ei osaa tulkita tyhjiä tai QGIS-sisäisiä tunnuksia.

**Ratkaisu:**

- Lisätty uusi metodi `validate_crs_selections()` `Prosessointi_dialog.py`-tiedostoon.  Se tarkistaa molemmat CRS-valitsimet ennen prosessoinnin käynnistystä ja lisää selkeän suomenkielisen virheviestin, jos kummassakaan valitaan muokattu koordinaattijärjestelmä:
  *"Koordinaattijärjestelmät (…): Muokattu koordinaattijärjestelmä ei ole tuettu. Valitse ennalta määritetty koordinaattijärjestelmä (esim. EPSG:3067, EPSG:3878, EPSG:3879)."*
- `validate_all_inputs()` kutsuu nyt `validate_crs_selections()` vaiheena 5, joten virhe näytetään käyttäjälle normaalin validointidialogissa ennen kuin exe käynnistyy.
- `collect_all_ui_values()` käyttää nyt WKT-fallbackia: jos `authid()` on tyhjä tai `USER:`-alkuinen, `desired_crs`/`original_crs`-arvoiksi kirjoitetaan `toWkt()` tyhjän merkkijonon sijaan, jolloin pyproj voi yrittää tulkita CRS:n WKT-muodosta.
