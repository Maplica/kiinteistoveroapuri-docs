# Patchnote — 2026-06-12 #1

## Overview

This patch replaces the old **automatic report classification logic** with a **user-controlled reporting workflow** based on explicit per-item toggles. Report inclusion is no longer inferred from origin layer names or edited field suffixes. Instead, the user decides in the item edit dialog exactly which reports each item should be written to.

The patch also modernises the report-generation UX:

- item reporting choices are now shown as **switches** instead of plain checkboxes
- the switches are placed in a **horizontal row at the bottom** of the edit dialog
- the main **Luo tiedosto** window now lists the new report outputs explicitly
- each report choice in **Luo tiedosto** now has a **hover description tooltip**
- building add/remove reports for Verohallinto are now split into **two separate CSV files**
- session save/load now preserves all reporting toggle states

The current implementation writes **CSV files only**. The root-level XLSM templates are not used in this workflow.

---

## Architecture Change

### Old model removed: inferred routing from layer names and edit sides

The previous system used two inference mechanisms:

1. `_LAYER_BUCKET_MAP`
   - routed items based on `origin_layer_name`
2. `_entry_edit_sides()` + `_collect_report_entries()`
   - routed items into `kunta` / `vero` buckets based on edited `__kunta`, `__vero`, and `__added` fields

This logic has now been removed from the report generation flow.

### New model: `report_toggles` is the single source of truth

Every item in `_added_features_store` now contains:

```python
'report_toggles': {
    'add_to_verottaja': False,
    'remove_from_verottaja': False,
    'add_to_kunta': False,
    'remove_from_kunta': False,
    'rekisterikorjaus_vero': False,
    'rekisterikorjaus_kunta': False,
}
```

Report generation reads only these toggle values when deciding where an item should be written.

This means:

- multiple toggles are allowed
- one item may appear in multiple reports
- report routing is deterministic and user-controlled
- the old fallback/inference behavior is gone

## New Reporting UX

### AddedFeatureEditDialog: switch-based reporting controls

The added-item edit dialog now contains a new bottom section:

**Ilmoitustyyppi**

Instead of vertical checkboxes, the dialog now shows **switch-style controls** arranged left-to-right in a horizontally scrollable row.

Each switch is rendered as:

- label on top
- visual toggle below
- compact card-like cell in the row

#### Rakennus switches

| Toggle key | Label shown in dialog |
|---|---|
| `add_to_verottaja` | Lisää rakennus verotietoihin |
| `add_to_kunta` | Lisää kunnan rekisteriin |
| `remove_from_verottaja` | Poista verottajalta |
| `remove_from_kunta` | Poista kunnan rekisteristä |
| `rekisterikorjaus_vero` | Rekisterikorjaus verotiedot |
| `rekisterikorjaus_kunta` | Rekisterikorjaus kunnan rekisteri |

#### Maapohja switches

| Toggle key | Label shown in dialog |
|---|---|
| `add_to_verottaja` | Lisättävä rakennuspaikka |
| `remove_from_verottaja` | Poistettava rakennuspaikka |
| `rekisterikorjaus_vero` | Rekisterikorjaus verottajan tiedot |
| `rekisterikorjaus_kunta` | Rekisterikorjaus kunnan tietokanta |

### Mandatory selection

The edit dialog OK button is now disabled until **at least one** reporting switch is enabled.

### Conflict handling

Conflicting toggle pairs are enforced immediately in the UI:

| Conflict pair | Behavior |
|---|---|
| `add_to_verottaja` vs `remove_from_verottaja` | Activating one turns the other off |
| `add_to_kunta` vs `remove_from_kunta` | Activating one turns the other off |

This applies to both dialog categories where relevant.

## `Luo tiedosto` Window Changes

### New basic info field: `Kuntanumero`

The report generation dialog (`ReportOptionsDialog`) now contains a new input field:

- `Kuntanumero`

This is used in the Maapohja CSV outputs and is required when generating:

- `Maapohja – Lisättävät rakennuspaikat`
- `Maapohja – Poistettavat rakennuspaikat`

If either of those report types is selected and `Kuntanumero` is empty, the dialog cannot be accepted.

---

### Report choices are now explicit and documented by tooltip

Each checkbox in **Luo tiedosto** now has:

- a clearer description of what the output is for
- a tooltip describing:
  - whether the report goes to Verohallinto or is internal
  - which item toggles feed that report
  - the exact output filename pattern
  - the exact output CSV columns

The tooltip styling has been customised to match the rest of the plugin UI.

## CSV Population Functions

The logic has been split into dedicated report-building functions in `util/reportingutils.py`.

### New public functions

| Function | Purpose |
|---|---|
| `populate_puuttuvat_rakennukset_csv()` | Building Verohallinto add/remove CSV rows |
| `populate_rakennuspaikat_uudet_csv()` | Maapohja "uudet" CSV |
| `populate_rakennuspaikat_poistettavat_csv()` | Maapohja "poistettavat" CSV |
| `populate_rekisterikorjaus_verottaja_csv()` | Wrapper around existing field-diff logic for `side='vero'` |
| `populate_rekisterikorjaus_kunta_csv()` | Municipality field-diff + action rows in one CSV |

Each function returns:

```python
(DataFrame, list[str] validation_errors)
```

This allows:

- rows with valid data to be written
- rows with missing required data to be skipped
- user-visible warnings to be shown in the final summary dialog

## Validation Behavior

### Dialog-level validation

The **Luo tiedosto** dialog now enforces:

- at least one report type selected
- output directory selected
- `Kuntanumero` required when Maapohja `uudet` or `poistettavat` outputs are selected

### Row-level validation

Each populate function validates the minimum required fields before adding a row.

If a row cannot be written, it is skipped and a warning string is returned.

Examples of current validation behavior:

- Rakennus Verohallinto CSV: skips rows missing `Pysyvä rakennustunnus__vero` / fallback PRT
- Maapohja CSVs: currently require `Kiinteistotunnus`

The final report dialog now shows three independent sections when relevant:

- `Luotu:` — files written successfully
- `Ohitetut rivit (puuttuvat kentät):` — validation warnings
- `Virheet:` — hard write failures / exceptions

## Save / Load Changes

### Session schema extension

Saved session items now include:

```json
"report_toggles": {
  "add_to_verottaja": false,
  "remove_from_verottaja": false,
  "add_to_kunta": false,
  "remove_from_kunta": false,
  "rekisterikorjaus_vero": false,
  "rekisterikorjaus_kunta": false
}
```

### Backward compatibility

When loading older session JSON files that do not contain `report_toggles`:

- the file loads successfully
- all toggles default to `False`
- no migration error is raised
- the user can open the item dialog and select report toggles manually
- saving again writes the new structure automatically

## Notes / Known Limitations

1. The current workflow writes **CSV files only**.
   - The root-level XLSM templates are not used.

2. The building Verohallinto add and remove CSVs currently share the same column structure.
   - They are separated only by output file and filtering logic.

3. `Kuntanumero` is intentionally user-entered in the report dialog.
   - It is not resolved from `__vero` data.

4. The item edit dialog enforces at least one reporting toggle before OK.
   - This means every edited item must explicitly declare a reporting purpose.

## Result

The reporting system is now:

- explicit instead of inferred
- easier to understand for end users
- easier to extend with new report types
- more maintainable in code
- safer for session persistence and replay
- clearer in the `Luo tiedosto` UI through direct labels, file naming, and tooltips
